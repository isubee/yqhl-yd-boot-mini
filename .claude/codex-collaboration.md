# Codex 协作规则(项目级)

本文件规定 Claude Code 与 Codex 在本项目内的分工协作方式,
与项目根 `CLAUDE.md` 配合使用。

> 全局协作原则参见 `~/.claude/CLAUDE.md`。
> 本文件提供**项目无关的协作机制**:调用模板、审查流程、回退策略、双通道选择。
> 项目特有的技术栈约束、规范、禁区由项目根 `CLAUDE.md` 提供,
> 本文件通过「约束加载」机制引用,不内嵌任何具体技术细节。

---

## 1. 分工速查

| 任务类型 | 执行者 |
|---|---|
| 新增模块、文件、类、组件等成段代码 | **Codex** |
| 跨多文件的修改、重构、迁移 | **Codex** |
| 复杂逻辑实现(算法、状态机、业务编排) | **Codex** |
| Bug 修复(影响 >20 行) | **Codex** |
| 测试代码生成 | **Codex** |
| 改一两行配置、常量、注释 | Claude Code |
| 解释代码、回答疑问、写文档 | Claude Code |
| 规划、需求澄清、任务拆分 | Claude Code |
| 跑构建/测试命令、读日志、git 操作 | Claude Code |
| **代码审查(任何来源的代码)** | **Claude Code(不外包)** |

判断标准:**产出是"成段新代码"或"跨多文件改动"→ Codex;否则 Claude Code**。

## 2. 约束加载(关键机制)

调用 Codex 时,Claude Code 必须**主动收集并传入项目约束**,因为 Codex
不会自动读取项目的 `CLAUDE.md` 或文档。

### 收集约束的来源(按优先级)

1. **项目根 `CLAUDE.md`** —— 主要的项目规范、技术栈、禁区
2. **`.claude/` 目录下的项目级文档** —— 补充规范、风格指南
3. **当前对话中用户给出的临时约束** —— 本次任务专属
4. **代码库中已有的同类实现** —— 最强的"约束",见下文

### 「参考已有实现」原则

文档约束有上限,**最强的对齐手段是让 Codex 模仿已有代码**。
每次调用 Codex,Claude Code 应当先用 Glob/Grep 找到 1-2 个同类实现,
在 prompt 中指向它们的路径,让 Codex 阅读并模仿其风格、结构、命名。

例如:
- 新增模块 → 指向某个最近、最规范的现有模块
- 新增组件 → 指向项目中风格最对的同类组件
- 新增测试 → 指向覆盖率高的已有测试文件

**没有"参考实现"指引,Codex 容易输出"通用最佳实践"风格的代码,
与项目实际风格不符。**

### 约束块的组装方式

每次调用 Codex,prompt 中必须包含一段「项目约束」,内容由 Claude Code
从上述来源**实时摘录**而成。建议格式:

```
【项目约束】

技术栈与版本:
[从项目 CLAUDE.md 摘录]

代码规范与命名:
[从项目 CLAUDE.md 摘录]

目录结构与分层:
[从项目 CLAUDE.md 摘录]

禁止修改的范围:
[从项目 CLAUDE.md 摘录]

参考已有实现:
[Claude Code 用 Glob/Grep 现场定位的 1-2 个文件路径]
```

不要把整份 `CLAUDE.md` 原样塞给 Codex,**摘录任务相关部分即可**,
避免 prompt 过长稀释关键约束。

## 3. 标准调用模板

Claude Code 通过 codex MCP 工具调用 Codex 时,按以下三种模板之一组织 prompt。
具体业务细节由用户在对话中提供,Claude Code 填充进模板。

> 显式调用(用户打 `/codex:rescue` 等命令)的场景见第 9 节。

### 模板 A:新建(添加新文件 / 新模块 / 新组件)

适用场景:产出是若干个新文件,基本不动现有代码。

```
任务:[一句话描述要新建什么]
工作目录:[传目标子目录的绝对路径,而非项目根,以最小化 Codex 的影响范围]
sandbox: workspace-write

【项目约束】
[Claude Code 按第 2 节方法摘录组装]

参考已有实现:
请先阅读 [具体文件路径] 作为参考。
你的代码风格、注解/装饰器、命名、目录结构必须与之保持一致。

业务需求:
[Claude Code 把用户描述填入,字段/接口/行为讲清楚]

需要产出的文件清单:
- [路径 1]:[作用]
- [路径 2]:[作用]
- ...

输出格式:每个文件一段完整代码,文件之间用分隔线
```

### 模板 B:精修(在已有文件上小范围改动)

适用场景:1-3 个文件,改方法、加字段、调整逻辑。

```
任务:[一句话描述]
工作目录:[项目根或更精确的子目录]
sandbox: read-only(返回 unified diff,Claude Code 落盘)

变更点:
- 文件:[路径]
- 位置:[函数 / 类 / 行号]
- 改什么:[具体描述]

【项目约束】
[摘录组装]

本次任务专属约束:
- 不修改函数/方法签名(向后兼容)
- 不触碰其他函数或文件
- 保留原有的事务/锁/异常处理边界
- 不顺便重构、格式化、优化邻近代码

输出格式:unified diff patch
```

### 模板 C:Bug 定位与修复

适用场景:有明确的报错、异常或行为偏差。

```
任务:定位并修复以下 Bug
工作目录:[项目根]
sandbox: read-only(先返回方案与 patch,确认后再落盘)

现象:
[报错堆栈、日志、复现步骤、期望行为 vs 实际行为]

可能涉及的文件:
[Claude Code 用 Glob/Grep 先初步定位后填入,最多 3-5 个]

【项目约束】
[摘录组装]

修复约束:
- 修根因,不修症状
- 不顺便重构其他代码
- 修复后补一个能稳定复现该 Bug 的回归测试

输出:
1. 根因分析(3-5 句话)
2. 修复方案(分点说明)
3. unified diff patch(包含修复 + 测试)
```

## 4. Codex 产出后的强制审查流程

**无论 Codex 是通过 MCP 自动调用还是通过 `codex-plugin-cc` 命令显式调用,
Codex 每次返回代码后,Claude Code 必须完成以下审查**,
任一项不通过 → 让 Codex 修(不要 Claude Code 自己改),通过后才落盘 + commit。

### 通用审查清单

**项目规范层(以项目 CLAUDE.md 为准)**
- [ ] 技术栈/版本/语法符合项目要求
- [ ] 包路径/目录结构正确
- [ ] 命名规则、文件后缀、命名空间符合约定
- [ ] 注解/装饰器/导入符合项目风格

**禁区层**
- [ ] 没有修改项目 CLAUDE.md 标注的"禁止修改范围"
- [ ] 没有动构建/依赖配置(除非任务明确要求)
- [ ] 没有把示例密钥/凭据/测试数据提取为常量

**质量层**
- [ ] 异常处理符合项目惯例
- [ ] 日志/可观测性按项目规范
- [ ] 输入校验在正确的边界层
- [ ] 没有未使用的导入、变量、参数

**对齐层**
- [ ] 风格与「参考已有实现」对齐(不是"通用最佳实践"风)
- [ ] 没有未要求的抽象层、工具类、配置项("聪明地"加东西)
- [ ] 没有顺手的 drive-by 重构

**任务范围层**
- [ ] 改动范围与任务描述一致,没有越界
- [ ] 没有删除/重写"看起来更好但不该动"的现有代码
- [ ] 测试(若涉及)只覆盖本次改动,不大面积重写已有测试

### 项目特有清单

如果项目有专属的审查清单(在项目 `CLAUDE.md` 或 `.claude/` 下),
**也必须走一遍**。例如:权限注解、多租户、数据脱敏等。

### 审查产出物

审查完成后,Claude Code 应当向用户报告:

- ✅ 通过的项
- ⚠️ 让 Codex 修过的项(简要说明问题)
- 📋 主观判断/可选优化(留给用户决定是否调整)

## 5. sandbox 策略

| 场景 | sandbox | 原因 |
|---|---|---|
| 新建多文件(模板 A) | `workspace-write` | 直接落盘高效,审查在落盘后做 |
| 精修(模板 B) | `read-only` | 返回 patch,Claude Code 审完再决定是否应用 |
| Bug 定位(模板 C) | `read-only` | 先看方案,确认无误后再落盘 |
| 涉及禁区的请求 | **拒绝调用** | 直接告知用户,不进入 Codex |

**关键安全做法**:`cd` 参数尽量传**任务相关的子目录**而非项目根。
这样 Codex 物理上就动不到项目里 Claude Code 标注的禁区。

例如:
- 新模块开发 → `cd` 设到 `<项目根>/src/modules/<新模块名>` 或类似的子目录
- 仅当任务跨多个目录时,才把 `cd` 提到更高层级

---

## 6. 反盲信原则

Codex 返回的代码 **不是终稿**,Claude Code 必须保留独立判断:

- 对照项目约束与审查清单 **逐条核对**,不自动通过
- 发现 Codex 引入未要求的抽象层、工具类、配置项 → 让它删
- 发现 Codex 用了"看起来更好但与项目风格不符"的写法 → 让它对齐
- Codex 的根因分析或方案有疑点 → 提出反问,要求重新论证
- **协作不是替代思考**:Claude Code 不应该当甩手掌柜

判断 Codex 输出可信度的几个信号:

| 信号 | 含义 |
|---|---|
| 代码风格与"参考已有实现"高度一致 | 喂养充分,可信度高 |
| 引入了 prompt 里没要求的依赖、抽象、工具 | 警惕,大概率多余 |
| 改动范围超出任务描述 | 警惕,可能 drive-by |
| 测试用例覆盖了任务核心路径 | 良性信号 |
| 大段重写了未要求改动的代码 | 拒绝,要求收窄范围 |

## 7. 不调用 Codex 的场景

以下任务 Claude Code **直接处理**,不要调用 Codex(避免过度协作):

- 改 1-2 行配置文件、常量、字面量
- 修一个 typo、调一个注释
- 加一两行枚举值、错误码
- 解释代码、说明机制如何工作
- 写文档、ADR、commit message、README
- 执行构建/测试/git 命令、读日志
- 单纯定位代码位置(用 Glob/Grep)

判断锚点:**预估最终代码改动 < 20 行 → Claude Code 自己来**。

调 Codex 也要付出代价(网络往返、审查成本、上下文切换),
小事情自己做更快。

## 8. 与 slash 命令工作流的整合

如果项目使用 addyosmani/agent-skills 或类似的 slash 命令体系,
按以下方式整合 Codex:

| 命令类型 | 执行者 | Codex 介入点 |
|---|---|---|
| 规划类(spec/plan/brainstorm) | Claude Code | 不介入 |
| **构建类(build/implement)** | Claude Code 主导 | **介入**:成段代码委托 Codex |
| **测试类(test)** | Claude Code 主导 | **介入**:测试代码委托 Codex,验证由 Claude Code 跑 |
| **审查类(review)** | Claude Code | **绝对不介入**(守门角色不外包) |
| 简化类(simplify) | Claude Code 主导 | 可介入:方案 Claude Code 出,落地 Codex 做 |
| 文档类(docs/adr) | Claude Code | 不介入 |

**审查类命令绝对不外包给 Codex**,因为审查标准来自项目 `CLAUDE.md`,
而 Claude Code 是这份文件的"读者"。让 Codex 审等于让选手当裁判。

## 9. Codex 调用通道选择(双通道并存)

本项目可同时启用两种 Codex 调用通道,各有适用场景。Claude Code 应根据
任务特征选择通道,**不要把所有任务都套同一种**。

### 通道一:MCP 自动委托(默认主力通道)

**形式**:`codex mcp serve` 注册为 Claude Code 的 MCP 工具,Claude Code
自主判断并调用,prompt 完全由 Claude Code 控制。

**适用场景**:

- 日常 `/build` 流程中的单切片代码生成
- 模板 A / B / C 覆盖的所有任务
- 任务预估 < 5 分钟,适合同步等待
- 需要精确控制 prompt 内容(约束块、参考实现)

**优势**:

- Claude Code 控制每次调用的 prompt,约束加载机制(第 2 节)生效
- 自动委托,符合 Codex-First 原则,人不用每次显式触发
- 与本文件第 3 节调用模板完全配套

**这是日常开发的默认通道。**

### 通道二:`codex-plugin-cc` 显式委托(长任务/分支审查专用)

**形式**:OpenAI 官方 Claude Code 插件,提供 `/codex:*` 系列命令,
**由用户显式触发**。

**适用场景与命令**:

| 场景 | 推荐命令 |
|---|---|
| 任务预估 > 5 分钟,需后台跑 | `/codex:rescue --background <任务描述>` |
| 后台任务进度查询 | `/codex:status` |
| 取回完成的后台任务结果 | `/codex:result` |
| 取消运行中的后台任务 | `/codex:cancel` |
| 整个分支/PR 的全面 Codex 审查 | `/codex:review --base <ref>` |
| 关键任务用更高规格的模型 | `/codex:rescue --model <x> --effort high` |
| 续跑上次中断的 Codex 任务 | `/codex:rescue --resume` |
| 检查 Codex 环境是否就绪 | `/codex:setup` |

**重要约束**:

- **审查规则不变**:即使通过插件命令调用 Codex,产出后 Claude Code
  仍然必须走第 4 节的审查清单。**插件不豁免审查义务**。
- **`/codex:review` 是 Codex 自审产出,不能替代** Claude Code 对 Codex
  产出的审查。两者层次不同。
- **review gate 功能保持关闭**:插件的 Stop hook 自动审查 gate 与本项目
  的"审查不外包"原则冲突,且容易导致 Codex/Claude 互审死循环、
  快速消耗 quota。除非有明确理由,不启用。
- **后台任务也要审**:`--background` 完成后,通过 `/codex:result` 取回的
  代码同样需要 Claude Code 走审查流程,落盘前不可省略。

### 通道选择判断流程

Claude Code 收到代码任务后:

1. 是否琐碎(< 20 行 / 单文件配置)?→ **Claude Code 自己做**(第 7 节)
2. 是否预估 > 5 分钟、可后台跑?→ 提醒用户用 `/codex:rescue --background`
3. 用户是否显式打了 `/codex:*` 命令?→ 走通道二,审查不变
4. 否则 → **走通道一(MCP 自动委托)**,按第 3 节模板

### 配置要点

- 两种通道**可以并存**,互不冲突,共享同一个本地 Codex CLI
- 默认配置即可,无需特殊调整
- 关闭插件的 review gate(具体配置参见 `codex-plugin-cc` 文档)

## 10. 协作失效时的回退

出现以下情况,Claude Code 应当**停止调用 Codex**(MCP 通道),
直接独立完成任务并告知用户:

- Codex MCP 不可达(`codex: not connected` 或调用超时)
- Codex 连续两次产出明显违反项目约束(说明上下文喂养有问题,继续也是浪费)
- 任务实际比预估琐碎(<20 行最终改动)
- 任务无法在不触碰禁区的情况下完成
- 用户明确要求"这次不要用 Codex"

回退到独立模式时,Claude Code 仍然遵守:
- 项目根 `CLAUDE.md`
- 全局 `~/.claude/CLAUDE.md`
- 本文件第 4 节的审查清单(对自己的产出也要审)

**注**:用户显式打 `/codex:*` 命令的场景不走这套回退,因为是用户主动选择
通道二;若插件本身报错,提示用户检查 `/codex:setup`。

## 11. 调用前的自检清单

每次准备**通过 MCP 调用** Codex 前,Claude Code 内部走一遍:

- [ ] 这真的是非琐碎代码任务吗?(否则别调)
- [ ] 是否更适合走通道二(后台长任务、跨分支审查)?(若是,提醒用户)
- [ ] 我已经从项目 `CLAUDE.md` 摘录了相关约束吗?
- [ ] 我已经用 Glob/Grep 定位了「参考已有实现」吗?
- [ ] 任务描述足够具体吗?(字段、接口、行为讲清楚了吗)
- [ ] sandbox 选对了吗?(write 还是 read-only)
- [ ] `cd` 传的是合适的子目录吗?(不是项目根)
- [ ] 这次调用涉及禁区吗?(若是,直接拒绝调用)

全部 ✓ 才调用。任何一条 ✗ 先解决再调。

通道二(用户显式命令)不走此清单,但**审查清单(第 4 节)对两种通道
同等适用**。
